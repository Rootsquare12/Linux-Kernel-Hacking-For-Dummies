# 0-2. 운영체제 해킹이 무엇일까?

## 운영체제의 취약점

앞서 우리는 운영체제가 컴퓨터의 하드웨어와 시스템 자원을 관리하는 **소프트웨어**라고 배웠습니다. 따라서, 이 소프트웨어 역시 응용 프로그램들처럼 버그가 발생할 수 있고, 보안 취약점이 있다면 해커에게 악용될 수 있습니다.

그러나 일반적인 응용 프로그램들과는 달리, 운영체제는 **시스템의 최고 권한(커널 영역)에서 작동하는 유일한 주체**입니다. 이 차이 때문에 운영체제 결함은 시스템에 심각한 장애 또는 피해를 유발합니다.

대표적인 예시는 다음과 같습니다.

- 메모리 누수, 교착상태(Deadlock), 경쟁 상황(Race condition) 등으로 인해 시스템의 성능이 저하 또는 마비됩니다.
- 장애로 인해 커널이 멈추거나 오류를 일으킬 경우, 보호 조치로 즉시 시스템을 정지시킵니다.(이것을 커널 패닉(Kernel Panic)이라 부르며, Windows의 블루스크린(BSOD)과 같은 의미입니다.)
- 만약 운영체제에서 보안 취약점이 악용될 경우, **공격자는 커널 권한에서 시스템 전체를 제어할 수 있게 됩니다.**

## 운영체제의 보안 취약점이 악용된다면?

일반적으로 워게임이나 CTF 등으로 기초 시스템 해킹을 입문하신 분들이면,취약점을 익스플로잇 할 때 최종 목표로 다음을 가장 먼저 떠올릴 것입니다 : `system("/bin/sh")`

실행하면 운영체제와 상호작용하는 인터페이스인 셸(Shell)을 획득하며, 일반적으로 시스템 해킹의 성공으로 인정합니다. 하지만 명심하세요. **이 셸은 취약점이 발생한 해당 프로그램이 가진 제한된 권한(user, 사용자)을 갖습니다.** 응용 프로그램들은 사용자 영역에서 동작하기 때문입니다.

그렇다면 커널 영역에서 동작하는 운영체제가 해킹을 당한다면 어떻게 될까요? 공격자는 아래와 같은 치명적인 행위를 할 수 있습니다.

1. **임의 메모리 읽기 및 쓰기(AAR/AAW)** : 커널 메모리 등 시스템의 어떤 위치든 마음대로 데이터를 읽고 쓰거나, 운영체제의 실행 흐름을 통째로 조작합니다.
2. **보안 장치 무력화** : 시스템의 모든 권한을 통제하므로, 컨테이너(Docker 등)나 샌드박스 같은 강력한 보안 격리 장치들을 무시하고 탈출할 수 있습니다.
3. **권한 상승(Local Privilege Escalation, LPE)** : 일반 사용자의 권한을 임의로 조작하여, 시스템의 최고 관리자 권한인 **루트(root) 권한**을 손쉽게 획득합니다. **커널 해킹의 가장 대표적인 목표입니다.**
4. **시스템 파괴(DoS)** : 고의로 커널 오작동을 유발하여 시스템을 크래시(Crash)시킵니다.

비유하자면, 응용 프로그램 해킹이 어떤 건물의 방 하나를 털어가는 것이라면, 커널 해킹은 **해당 건물 전체를 탈취하는 것**과 같습니다. 따라서 Linux 커널 해킹이란, 시스템의 가장 깊고 은밀한 곳까지 분석하고 제어하며, 취약점을 발굴해 내고 이를 패치하는 모든 과정을 의미합니다.

## 그래서 Linux 커널의 어디를 공격하는 건데?

앞서 Linux 운영체제가 기본적으로 모놀리틱 구조이지만, 필요할 때마다 커널 모듈(LKM)을 꽂아 넣을 수 있다는 사실을 기억하시나요? 이 뼈대를 바탕으로 커널 영역은 크게 두 가지로 나눌 수 있습니다.

|명칭|설명|
|---|---|
|주 커널(Main Kernel)|리눅스의 뼈대가 되는 핵심 코드입니다.<br>리누스 토르발즈를 비롯한 세계 최고 수준의 개발자들이 매일 코드를 검수하고, 수많은 자동화 도구들이 24시간 취약점을 감시합니다. 버그가 발견되면 즉각적으로 패치되기 때문에 이곳을 뚫는 것은 극도로 어렵습니다.|
|커널 모듈 (Kernel Module, LKM)|그래픽 카드, 와이파이 어댑터 등 다양한 장치를 위해 추가로 꽂아 넣는 코드들입니다. 수많은 하드웨어 제조사나 개인 개발자들이 작성하기 때문에 주 커널만큼 까다로운 검수를 거치지 않는 경우가 많습니다. **하지만! 이 모듈들도 일단 커널에 적재(Load)되면, 주 커널처럼 '커널 권한'을 갖게 됩니다.**|

비유하자면, 주 커널이 항상 경비가 삼엄한 하나의 큰 **요새**고, 커널 모듈(LKM)은 성으로 새로 들인 **외부 업체/용병**과도 같습니다. 이 용병들이 실수를 하거나(취약점) 악의를 가질 경우(악성 모듈), 일단 요새(커널) 안으로 들어온 이상 커널 공격에 활용할 수 있습니다!

따라서 우리는 이 책에서 해킹 실습을 진행할 때, 거대한 주 커널을 통째로 분석하는 대신 **취약하게 만들어진 실험용 '커널 모듈'들을 공격하며 취약점의 원리를 꿰뚫어 볼 것입니다.**

### 부록: 커널 취약점 통계와 CTF에서의 커널 해킹

미국 국립표준기술연구소(NIST)의 통계에 따르면, Linux 커널에서 발생한 실제 취약점(CVE)의 **60~70% 이상이 드라이버나 파일 시스템 같은 '모듈' 영역에서 발생합니다.** (리눅스 커널 전체 코드 중 주 커널이 차지하는 비중은 10%도 안 됩니다. 코드 양만 봐도 모듈에서 취약점이 터질 확률이 압도적으로 높습니다!)

또한, 해킹 방어대회(CTF)에서도 최신 리눅스 주 커널 자체의 취약점을 문제로 내는 것은 현실적으로 무리가 있습니다. 최신 커널의 진짜 취약점(0-day)은 어마어마한 가치를 지니기 때문입니다.

따라서 출제자들은 **'일부러 취약점을 숨겨둔 커널 모듈'을 만들어 시스템에 적재해 두고 문제를 출제합니다.** 참가자(여러분)의 최종 목표는 일반 사용자 권한에서 익스플로잇 코드를 실행하여, **이 모듈의 취약점을 사용하여 자신의 권한을 최고 관리자(root)로 둔갑시키는 것(권한 상승)**이 됩니다. 이것이 바로 우리가 앞으로 배울 커널 해킹의 핵심입니다!
